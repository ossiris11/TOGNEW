# TOGOSHOL - фронтенд-аудит UX и дизайна

Дата аудита: 05.06.2026.

Охват: публичный лендинг, каталог, конфигуратор, модальные сценарии, мобильная адаптация, админский фронтенд. Backend-логику и API не меняем.

Проверка: `npm run build` проходит успешно. Аудит сделан по коду и CSS; отдельную визуальную проверку через Playwright/screenshot нужно добавить перед релизом.

## P0 - исправить до публичного запуска

- **Первый экран на мобильных может конфликтовать с фиксированным header.** В `Hero.css` на mobile у `.hero` верхний padding 54px, а header занимает около 78px. Нужно проверить 360-430px и увеличить безопасный отступ, чтобы заголовок и badge не уезжали под шапку.

- **Мобильное меню нужно довести до полноценного overlay-сценария.** Сейчас меню раскрывается внутри header (`Header.tsx`, `Header.css`), но нет закрытия по Escape/клику вне меню, нет body scroll lock, нет управления фокусом. Для пользователя это особенно заметно на телефоне: можно прокручивать страницу под открытым меню.

- **Плейсхолдеры отзывов нельзя показывать как публичный контент.** В `WhyTogoshol.tsx` при отсутствии реальных отзывов показываются "Слот под широкий скрин отзыва". Для релиза лучше скрывать блок отзывов, показывать честный empty-state для админа или выводить только реальные published-отзывы.

- **Сценарий покупки в каталоге слишком сложный.** Кнопка "Купить ПК" открывает модалку, где пользователь должен скопировать текст и потом выбрать канал. Нужно сделать один главный путь: "Написать по этой сборке" сразу с предзаполненным сообщением в VK/Telegram/Avito, а копирование оставить вторичным действием.

- **Модалки и drawer требуют accessibility-доработки.** `productSpecOverlay`, `checkoutOverlay`, `customPartsOverlay` закрываются по Escape и overlay click, но нет focus trap, возврата фокуса на кнопку открытия, начального фокуса на заголовок/закрытие и единых aria-состояний. Это критично для клавиатуры и мобильных скринридеров.

- **Формы заявок требуют настоящей валидации и понятных состояний.** В `CustomBuild.tsx` contact-поля принимают любой текст, ошибка одна общая, success не объясняет следующий шаг. Нужно добавить label, подсказку формата, валидацию телефона/Telegram/VK, disabled/loading, success с понятным "мы ответим..." и возможностью отправить снова.

- **Навигационные ссылки нужно синхронизировать с фактическими блоками.** В header ссылка "Отзывы" ведет на `#why`, где сначала преимущества, а отзывы ниже. "Под заказ" ведет на `#custom-parts`, которого нет в DOM и открывается только через JS-action. Лучше сделать отдельный якорь/кнопку для сборщика комплектующих и точный якорь для отзывов.

- **Карточка hero-сборки ведет себя нестабильно.** В `Hero.tsx` карточка является `article role="button"` с вложенной ссылкой, а активная карточка меняет ширину (`Hero.css`). На touch-экранах это может давать скачки горизонтального списка и неочевидное попадание по CTA. Нужен стабильный размер, явная кнопка выбора и отдельная CTA-зона.

## P1 - улучшить удобство и коммерческую ясность

- **Каталог: сделать фильтры более понятными на mobile.** Сейчас фильтры горизонтально скроллятся без явного признака продолжения. Добавить мягкую маску/градиент края, sticky toolbar или компактный bottom-sheet фильтров для телефонов.

- **Каталог: пересмотреть блок FPS.** `productFpsBox` показывает расчетный FPS и кнопку "?", которая на mobile скрывается. Нужно либо раскрывать методику/игры, либо заменить на честные сценарии "Full HD", "2K", "стриминг", "работа". Недоказанный FPS снижает доверие.

- **Карточки товаров визуально выбиваются из дизайн-системы.** Белая витрина внутри темного сайта выглядит эффектно, но спорит с dark premium tech. Стоит проверить единый стиль: меньше имитации маркетплейса, больше реального фото ПК, характеристик и статуса.

- **Добавить loading skeleton и error-state.** `useProducts`, reviews и комплектующие из API сейчас визуально не объясняют загрузку/ошибку. Для каталога, отзывов и кастомного сборщика нужны skeleton, "не удалось загрузить" и fallback CTA.

- **Сделать единый компонент контактов.** Header, mobile menu, footer/final CTA и checkout повторяют соцссылки вручную. Нужно вынести `SocialLinks`/`ContactButtons`, чтобы тексты, aria-label, tracking и порядок каналов не расходились.

- **Упростить конфигуратор.** В блоке много radiogroup, бюджет, готовые рекомендации, closestProducts, три CTA и форма. На mobile это длинный путь. Лучше сделать пошаговую компоновку: бюджет и задача, затем рекомендация, затем один главный CTA.

- **Сборщик из комплектующих сделать ближе к настоящему конфигуратору.** Сейчас это список select-полей. Для пользователя удобнее карточки категорий, выбранное состояние, предупреждения о несовместимости, подсчет "не хватает обязательных деталей" сверху и sticky summary на mobile.

- **FAQ и условия стоит усилить коммерческими деталями.** Добавить конкретику про гарантию, проверку комплектующих, сроки, оплату, самовывоз/доставку, работу с б/у деталями и возврат. Сейчас часть текста звучит общо.

- **Фото работ должны иметь fallback.** `SocialProof` полностью скрывается, если нет товаров с image. Лучше показать CTA "попросить фото конкретной сборки" или вывести избранные фото из админки.

## P2 - polish, доступность и производительность

- **Оптимизировать изображения.** `hero-pc` в production build весит около 1.32 MB. Нужны WebP/AVIF, responsive sizes, lazy для неhero-изображений и проверка LCP.

- **Пересмотреть шрифт и насыщенность.** Весь сайт использует Montserrat с весами 600-900. Для длинных характеристик и админки лучше добавить более читаемый text-font, например Inter, а Montserrat оставить для акцентов.

- **Уменьшить визуальный шум от glow/градиентов.** Текущая палитра сильно опирается на cyan/violet. Нужно добавить нейтральные поверхности, лучше разделить CTA-цвета и состояния, проверить контраст muted-текста.

- **Добавить явные focus-visible состояния всем интерактивным элементам.** Особенно header links, карточки hero, buttons в каталоге, details/summary FAQ, админские кнопки и select.

- **Убрать символы вместо иконок там, где они выглядят технически.** В каталоге используются `▣`, `◈`, `▤`, `◎`, `▥`, стрелки и `?`. Лучше заменить на понятные SVG/lucide-иконки или текстовые подписи.

- **Проверить отсутствие горизонтального overflow.** Особые зоны риска: hero carousel, catalog filters, product cards с длинными GPU/CPU, custom parts select, admin tables, metrics chart.

- **Добавить визуальную регрессию.** Минимум скриншоты 360, 390, 768, 1024, 1440 для главной страницы, checkout modal, spec drawer, custom parts modal, admin dashboard.

## Админский фронтенд

- **Добавить подтверждение для архивирования.** В товарах, комплектующих и отзывах кнопки "Архив"/"Скрыть" срабатывают сразу. Нужен confirm-modal или undo-состояние.

- **Улучшить мобильную админку.** Sidebar становится сеткой из двух колонок, таблицы превращаются в одноколоночные карточки. Для реальной работы на телефоне лучше sticky top tabs, компактные карточки заявок и быстрые действия.

- **Добавить empty/loading/error состояния во все вкладки.** Сейчас при ошибке API часть экранов может выглядеть пустой. Админу нужно видеть, что именно не загрузилось и что делать.

- **Формы админки требуют сохраненных/несохраненных состояний.** Нужны disabled на время сохранения, inline-ошибки, reset confirmation, превью изображения товара/отзыва и подсказки обязательных полей.

- **Дашборд сделать более сканируемым.** `metricsChart` зависит от title-tooltip и цвета. Добавить легенду, числовые подписи при hover/focus и короткие выводы: "заявок сегодня", "самый активный товар".

## Рекомендуемый порядок работ

1. Исправить mobile header/hero, меню, якоря и публичные плейсхолдеры отзывов.
2. Упростить покупку из каталога и довести модалки до корректной accessibility-модели.
3. Доработать формы заявок, состояния загрузки/ошибок и контактные CTA.
4. Провести mobile-polish каталога и конфигуратора.
5. Оптимизировать изображения, типографику и визуальную консистентность.
6. Отдельным проходом улучшить админку: подтверждения, loading/error, мобильные таблицы.
